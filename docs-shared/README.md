# docs-shared 共享文档目录

`docs-shared/` 是仓库内的共享文档入口，用来存放需要进入 Git 历史、可被多人查看和持续维护的文档。

这里的内容应当满足两个前提：

- 与当前项目协作直接相关
- 适合随代码一起评审、回溯和演进

## 目录用途

本目录主要用于沉淀以下类型的共享信息：

- 需要团队共同遵循的约定
- 需要保留历史的架构或实现决策
- 需要多人同步的运行说明、交付说明或协作背景

如果一份文档需要在代码评审中被看到，或者后续需要结合提交历史追踪变化，就应该优先考虑收录到这里。

## 与 `Docs/` 的边界

`docs-shared/` 和 `Docs/` 的用途不同：

- `docs-shared/`：仓库共享文档，纳入 Git，面向协作和长期维护
- `Docs/`：本地专用目录，不纳入 Git，适合个人记录和临时材料

如果某份内容最初写在 `Docs/` 中，但后续需要进入团队协作、评审或长期留档，应在整理后迁入 `docs-shared/`。

## 收录原则

建议收录到 `docs-shared/` 的内容通常具备以下特征：

- 需要多人阅读、讨论、评审或复用
- 需要与代码版本保持对应关系
- 需要为后续开发、排障或决策提供上下文
- 内容已经整理到适合共享的程度，而不是仅供个人临时使用

典型例子包括：

- 架构或目录调整后的共享说明
- 重要技术取舍的决策记录
- 需要跨角色同步的接口、运行或交付说明

## 不建议收录的内容

以下内容通常不建议直接放入 `docs-shared/`：

- 个人排查时的临时命令记录
- 只对本机环境有效的私有说明
- 尚未整理完成的草稿、碎片笔记或一次性备忘
- 不适合进入 Git 历史的个人材料或敏感内容

这类内容更适合先保留在个人工作区或 `Docs/` 中，待整理为可共享版本后再决定是否纳入本目录。

## 使用建议

如果后续需要在 `docs-shared/` 中新增内容，建议优先做到：

- 标题明确，能直接说明文档用途
- 内容基于当前仓库现状，不写无法落地的占位说明
- 结构稳定后再纳入 Git，避免把大量临时草稿直接带入共享目录

当目录下新增新的共享文档类别时，也应同步更新本 README，确保它继续承担共享文档入口的职责。

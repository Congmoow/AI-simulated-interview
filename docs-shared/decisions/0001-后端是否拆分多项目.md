# 0001 后端是否拆分多项目

## 状态

已接受

## 背景

当前后端仍以 `backend/src/AiInterview.Api` 单项目承载 API、服务注册、业务服务、仓储和基础设施接线。  
随着功能增加，`Program.cs` 的注册逻辑已经明显变重，但业务边界、依赖方向和多人协作方式还没有稳定到值得立即拆成多项目。

本轮结构治理的目标是先把组合根拆薄、把注册职责分组，而不是提前引入 `Domain / Application / Infrastructure / Api` 多项目结构。

## 决策

本轮不拆分后端多项目。  
当前先通过以下方式降低复杂度：

- 将认证注册抽到 `AuthenticationExtensions`
- 将 Swagger 注册抽到 `SwaggerExtensions`
- 将持久化注册抽到 `PersistenceExtensions`
- 将业务服务注册抽到 `ApplicationServicesExtensions`
- 将外部依赖注册抽到 `ExternalDependenciesExtensions`

也就是说，先整理组合根，再观察边界是否稳定，而不是为了“看起来更标准”提前拆出空壳项目。

## 现在不拆的原因

- 业务仍在快速演进，直接拆多项目会带来大量命名空间、引用关系和目录迁移成本
- 当前主要痛点是 `Program.cs` 过重，而不是项目边界已经清晰到必须物理拆分
- 过早拆分会增加调试、迁移、测试和新成员理解成本
- 现阶段先做依赖注册分组，已经可以显著降低组合根复杂度

## 触发重新评估的条件

当出现以下任一情况时，应重新评估是否拆分多项目：

- `Services/` 与 `Repositories/` 继续显著膨胀，单项目目录冲突频繁出现
- 测试经常需要跨层 mock，说明边界已经开始不清晰
- 多人并行开发时，API 项目目录成为高冲突区域
- 业务规则已经稳定，且存在明确的领域层、应用层、基础设施层分工

## 候选目标结构

若未来决定拆分，可优先考虑：

- `backend/src/AiInterview.Api`
- `backend/src/AiInterview.Application`
- `backend/src/AiInterview.Domain`
- `backend/src/AiInterview.Infrastructure`

## 影响

短期影响：

- `Program.cs` 会明显变薄
- 服务注册职责更清晰
- 行为保持不变，迁移风险可控

中期影响：

- 后续如果决定拆多项目，可以基于已经拆开的注册职责继续推进
- 团队可以先围绕“边界何时成熟”形成共识，而不是把拆分当作默认正确答案
